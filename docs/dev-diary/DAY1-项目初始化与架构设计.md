## 一、项目缘起

一直想做一个有意义的项目，把中国古典诗词和现代技术结合起来。苏轼是我非常喜欢的诗人，他一生的轨迹跌宕起伏，诗词作品与地点有着紧密的联系。于是"苏轼人生诗词地图"这个想法就诞生了。

**项目愿景**：创建一个交互式的诗词地图网站，通过苏轼的诗词作品串联起他的人生轨迹，让用户体验"跟着苏轼去旅行"的文化之旅。

---

## 二、技术选型

### 前端技术栈

| 技术 | 选择 | 理由 |
|------|------|------|
| 框架 | React 18 + TypeScript | 类型安全，生态成熟 |
| 构建工具 | Vite | 快速开发，按需编译 |
| UI 组件 | shadcn/ui | 美观现代，基于 Radix UI |
| 地图 | 高德地图 JS API 2.0 | 国内地点数据最准确 |
| 样式 | Tailwind CSS | 原子化 CSS，高效开发 |

### 后端技术栈

| 技术 | 选择 | 理由 |
|------|------|------|
| 框架 | FastAPI | 异步高性能，自动文档 |
| 数据库 | SQLite → PostgreSQL | 开发轻量，生产可靠 |
| ORM | SQLAlchemy 2.x | 成熟稳定，支持异步 |
| 爬虫 | httpx + BeautifulSoup4 | 轻量快速 |

### 部署方案

- 前端：Vercel
- 后端：Railway
- 数据库：PostgreSQL (Railway 托管)

## 四、数据库模型设计

### 核心模型

**诗词表 (Poetry)**
- 标题、内容、朝代、作者
- 创作年份、时期、题材
- 注释、译文、背景

**地点表 (Location)**
- 现代名称、古代名称
- 经纬度坐标
- 地址、省份、城市
- 景点信息、图片

**关联表 (PoetryLocation)**
- 诗词 ID、地点 ID
- 关联类型（创作地/提及地）
- 置信度（确定/推测/存疑）

## 五、第一天踩的坑

### 坑 1: 地图 API 密钥配置

一开始不知道高德地图需要配置安全密钥，导致地图一直无法加载。后来在官方文档找到配置方法：

```javascript
window._AMapSecurityConfig = {
  securityJsCode: import.meta.env.VITE_AMAP_SECURITY_CODE,
};
```

### 坑 2: 循环依赖问题

在定义 `PoetryLocation` 和 `Location` 的关系时，出现了 SQLAlchemy 的循环依赖错误。解决方法是使用 `back_populates` 而不是 `backref`，并且确保两边都正确配置。

### 坑 3: 异步 SQLAlchemy 的用法

FastAPI 使用异步 Session，一开始用惯了同步的 `.all()` 和 `.scalar()`，忘记加 `await`，导致各种奇怪的报错。
